## 【议题】字段统一：`premiumUSDC_6d`（整数） vs `premiumUSDC`（number）

**你提出的 4 步**（前端改名→后端对齐→Toast 提示→手动下单验证）本质是在**让前后端字段名一致**，避免 400。但这里有**单位与类型**的安全取舍，建议先定规则再动手：

### 这是什么

- 交易金额在系统内有两种表达：
  - `premiumUSDC_6d`：**以 micro‑USDC 的整数**传输（0.01 USDC = 10000）；
  - `premiumUSDC`：**以小数**传输（0.01）。
- 你的 4 步倾向于统一到 `premiumUSDC`（number）。

### 风险对比

- **使用 ****************************************************************`number`**************************************************************** 小数**：
  - JS 浮点（IEEE‑754）可能出现 `0.1 * 1e6 = 99999.999999…` 的舍入偏差；
  - 跨语言/跨库序列化差异，可能出现 `0.010000000000000002`；
  - 合约与监听器使用整数规则（1e‑6 步长）时，需要额外转换，易出错。
- **使用 ****************************************************************`*_6d`**************************************************************** 整数**：
  - 传输/持久化一致；
  - 与合约“可被 1e‑6 整除”的校验天然对齐；
  - 兼容 JSON（用字符串或安全范围内的整型数）。

### 三种方案

- **方案 A（推荐）**：统一使用 `premiumUSDC_6d`（请求与响应都为整数，API 用字符串承载），前端展示时再除以 1e6。
- **方案 B（你提议）**：统一 `premiumUSDC`（number），**但在前端拦截器**里强制：输入小数 → **精确转换**为 6 位整数后再发，后端仍以 `*_6d` 入库与校验。等价于“外部小数、内部整数”。
- **方案 C（过渡）**：兼容层同时接受 `premiumUSDC` 与 `premiumUSDC_6d`，写库统一 `*_6d`，**一周后移除**小数入口。

### 决策点

1. 是否坚持“**系统内只流通整数（\*\_6d）**”？
2. 若保留 `premiumUSDC` 小数入口，是否只在**前端边界**保留，入库一律 `*_6d`？
3. 错误码是否固定为 `ERR_SCHEMA_FIELD_MISMATCH`，并在消息体中附带 `expected: premiumUSDC_6d` 之类的提示？

### 验收标准（本议题通过的标志）

- 任意 3 笔“0.01/0.1/1 USDC”下单全部 201；
- 错发小数/错字段名时，Toast 展示“字段不匹配/单位不一致”，后端返回 `400` + 统一错误码 + 字段路径；
- 日志中打印入库值为**整数 6 位单位**，与展示值一致（展示值 = 入库 / 1e6）。

### 3) 替换内存桩为 SQLite

### 4) 监听器补确认与补扫

- **契约定义**：确认门槛 `CONFIRMATIONS=12`；两类任务：
  - **实时处理**：首次见到事件 → `paid_unconfirmed`（记录 `first_seen_block`）。
  - **周期补算**：每 N 秒重算 `confirms=tip - block + 1`，达标即迁移到 `paid_confirmed`。

### 1) RPC 主/备与速率限制（监听稳定性）

- **怎么判断 OK**：连续 12 小时无“丢块/超限”告警；主挂时 ≤1 分钟切到备；恢复后自动回切。
- **临时口径（上线优先）**：
  - **主 RPC**：Alchemy Base Mainnet（单主，先不配备节点）。
  - **速率上限**：≤ **3 req/s**。
  - **重试**：**3 次**（**250ms → 1s → 3s** 指数退避）。
  - **超时**：连接 **2s**；请求 **8s**。
  - **健康探测**：**60s** 间隔自检一次（仅日志告警，不自动切换）。

- **这是什么**：用现有单测/集成测 + 手工一单，确认 CRUD、幂等与持久性。

# 契约卡 v1 — 决定版（后端逻辑）参数确认纪要

本文档归档 2024-契约卡 v1 后端逻辑的临时参数决策，聚焦 Base 主网（chainId=8453）与原生 USDC（0x833589fCD6EDb6E08f4c7c32d4F71B54bDa02913）。如部署环境与本文档不一致，请在执行前更新参数并再次评审。

## A. 金额与字段

- **讨论焦点**：金额单位统一与字段命名是否与现有系统兼容，以及迁移节奏如何安排。
- **确认**：系统内部一律使用 micro-USDC（6 位整数），字段统一为 `premiumUSDC_6d`（字符串载体，内容为整数）。

- **错误码**：请求字段与单位不符时返回 `ERR_SCHEMA_FIELD_MISMATCH`，并在响应中附上期望字段/单位与示例值。
- **合理性评估**：统一存储单位避免浮点误差，兼容期提供缓冲；需同步校验各接口与批处理脚本均已适配。

## B. 环境变量与启动流程

- **讨论焦点**：启动依赖项是否完备、校验粒度是否足够，以及是否需要额外的安全提示。
- **确认**：`.env.sample` 必含以下键并在启动前校验，缺失即 Fail-Fast 退出：

## C. 数据库（SQLite）

- **讨论焦点**：SQLite 单实例模式能否满足吞吐需求，以及 WAL 配置在未来多实例扩展时的影响。
- **确认**：使用 `better-sqlite3`（进程级单例），数据库文件位于 `data/us-backend.db`。
- **配置**：启用 `journal_mode=WAL`、`synchronous=NORMAL`、`busy_timeout=3000ms`，自动创建目录并设置权限 600。
- **最小表集**：`users`、`sessions`（token 主键）、`orders`（`premium_usdc_6d` 为非空整数）、`api_keys`、`event_cursors`（topic 主键、记录 `last_block`）。
- **幂等要求**：`order_id` 唯一；重复创建返回 409 CONFLICT，响应体需携带已存在记录引用。
- **合理性评估**：单机 SQLite 适合当前 MVP 范围，WAL 设置兼顾写入性能与崩溃恢复；未来扩容需提前规划迁移脚本。
- 🟡 **待讨论/行动**：输出数据库迁移策略（含版本控制方案）并确认备份/恢复流程。

## D. 监听器（链上事件确认）

- **讨论焦点**：事件确认的状态迁移、回放策略，以及多确认门槛下的回滚处理。
- **确认**：
  - 首次捕获支付事件时写入 `paid_unconfirmed`，记录 `first_seen_block` 并更新 `event_cursors`。
  - 每 `LISTENER_POLL_INTERVAL_SEC` 轮询一次，计算 `confirms = tip - block + 1`，满足 `confirms ≥ CONFIRMATIONS`（默认 12）后迁移至 `paid_confirmed`。
  - 若配置 `REPLAY_FROM_BLOCK`，启动时回放历史事件直至当前链头。
  - 设置 `SAFE_DEPTH = CONFIRMATIONS + 2`（默认 14）以应对链重组；落入回滚区间的记录需退回并重放。
- **合理性评估**：按确认数分阶段处理能兼顾实时性与安全性；SAFE\_DEPTH 额外两层可抵御常见重组。
- 🟡 **待讨论/行动**：完成监听器状态机图，列出与数据库表设计的对应关系，并制定回放时的监控指标。

## E. 合约金额规则

- **讨论焦点**：链上步长限制与链下区间限制如何协同，以及是否需要链上增加阈值兜底。
- **确认**：链上 `amount` 必须满足 `amount % 1e6 == 0`。
- **阈值**：链下强制最小 0.01 USDC（10000）、最大 100 USDC（100000000）；链上仅校验步长。
- **合理性评估**：链下控制阈值可灵活调整，链上维持步长简化升级；需确保前后端共享同一常量源。
- 🟡 **待讨论/行动**：决定是否在合约或监听器加入异常阈值监控，并建立统一的常量发布流程。

## F. 证据与可复核策略

- **讨论焦点**：证据的真实性校验、规范化流程以及存储和稽核方式。
- **确认**：
  - 仅接受真实只读 API 数据，禁止样例/硬编码写入 `normalized_data`。
  - 规范字段映射：`exchange`、`account_id`、`symbol`、`order_id`、`side`、`leverage`、`margin_mode`、`liq_flag`、`timestamp_sec`、`filled_sz`、`avg_px`、`liq_px`、`reason`。
  - 采用 RFC 8785 JCS Canonical JSON，所有数值以字符串存储，剔除空字段。
  - `evidence_root = keccak256(UTF8(CanonicalJSON(normalized_data)))`。
  - 拉取失败标记为 `pending_evidence`，禁止生成占位数据。
- **合理性评估**：严格限制数据来源并执行 Canonical JSON 规范可保障可复核性；需关注不同交易所字段差异带来的映射复杂度。
